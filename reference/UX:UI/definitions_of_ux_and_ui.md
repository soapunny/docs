# UX / UI Made Simple

## 1) UX and UI in One Sentence

### UI (User Interface) = “The appearance of the screen + the buttons you operate”

- Buttons, text, colors, icons, menus, input fields, cards, spacing, animations, etc.
- **Things you can see and tap**

### UX (User Experience) = “The overall experience while using it”

- “Easy/hard,” “Fast/slow,” “Convenient/annoying,” “Trustworthy/scary”
- **The feelings and efficiency when using it**

✅ **Understanding by analogy**

- UI = The **steering wheel/buttons/dashboard** of a car
- UX = The **comfort/safety/stress** you feel when driving a car

> Even if the UI is beautiful, the UX can be bad, and even if the UX is good, if the UI isn’t good it can still be hard to use.

---

## 2) Why Is UX Important? (Examples)

### Example A: Convenience store kiosk

- The UI is clean and pretty, but
- The “Cancel” button is too small and hidden in a corner  
  → Users feel confused and annoyed  
  → **Bad UX**

### Example B: Delivery app

- The address input process has too many steps  
  → Users think, “This is annoying,” and close the app  
  → **Bad UX (drop-off)**

---

## 3) Eight Basic Building Blocks of UX (Simplified)

### 1) Research & Empathy

- Find out users’ needs and problems through interviews, surveys and observation

### 2) Problem Definition (What problem are we solving?)

- Summarize the research results and decide what discomfort users experience, what the problem/goal is

**Example**

- “Comparing prices at the market is too cumbersome—using chat rooms/flyers/word of mouth”
- Goal: “Let’s help them find the lowest price within 30 seconds”

---

### 3) Understanding the User (Know who uses it)

- Identify the user (who), context (when/where) and motivations/fears

**Example**

- International students/immigrants: English may be intimidating → Use simple words, provide support in their language
- Busy people: Too many steps cause drop-off → Let them access key features quickly

---

### 4) Information Architecture (IA) (Organizing menus/tabs)

- If you think of an app as a “bookshelf,” IA is deciding “where to place the books”

**Good example**

- Deals / Gas / Grocery / Profile  
  → You can immediately sense “where things are”

**Bad example**

- Daily info / Info sharing / Tips / Life tips / Help  
  → Similar menus make it confusing “where should I go?”

---

### 5) User Flow (Path to the goal)

- Check whether the user’s path to the goal is **short and natural**

**Example: “Check price” flow**

- Home → Category (market) → Search item (eggs) → View lowest price

**Bad flow example**

- Home → Forced login → Multiple terms agreements → Profile setup → Can search  
  → Many people leave in the middle

---

### 6) Wireframe & Prototype (Sketch → Clickable mock-up)

- Wireframe: the “pencil-drawn blueprint” of a screen
- Prototype: a working mock-up that you can actually click

**Example**

- First roughly decide button positions/menu structure, then
- If people get lost, quickly revise it

---

### 7) Usability (Is it easy for first-time users?)

- Even someone new should be able to use it right away

**Good example**

- You can browse without signing up
- If an input error occurs, it shows a friendly message
  - “Please enter numbers only for the price”

---

### 8) Validation (Testing) (Find problems/improvements by letting people actually use it)

- The creators might think “It’s obvious,” but
- Others might feel “I can’t find this”

**Example**

- Testing with just five people often reveals many “sticking points”
- Observe them in a real use scenario: “Where do they stop?”

---

## 4) Nine Basic Building Blocks of UI (Simplified)

### 1) Layout & Visual Hierarchy

- Making sure important things stand out first (size/weight/color/spacing)

**Example**

- If “View lowest price” is most important  
  → Make the button large and place it somewhere noticeable

**Bad example**

- The important button is small, gray and tucked in a corner  
  → People don’t see it

---

### 2) Typography

- Text must be easy to read

**Example**

- If the text is too small it hurts your eyes
- If line spacing is too tight it feels cramped

---

### 3) Color and Contrast

- Colors aren’t just for looks; they communicate meaning

**Example**

- Red: delete/danger/error
- Green: success/completed
- Gray: disabled (can’t click now)

**Bad example**

- Button color is similar to the background so it can’t be seen

---

### 4) Component Consistency (Unified rules)

- If buttons/input boxes/cards look different on each screen, users get confused

**Good example**

- The “Save” button always looks the same shape/color/size everywhere  
  → Users learn it quickly

**Bad example**

- In screen A the save button is blue
- In screen B it’s green
- In screen C it’s just an outline  
  → Users wonder “Is this the same function?”

---

### 5) State Design (Pressed/loading/error states)

- If nothing happens when you press a button, users feel uneasy  
  → They press again (multiple presses) → Risk of errors/duplicate payments/duplicate requests

**Good example**

- Show a loading indicator when the button is pressed
- Show “Saved successfully!” when done
- Show “Network error. Please try again.” when it fails

---

### 6) Progressive Disclosure

- Reduce cognitive load by showing only what’s needed at each stage instead of everything at once
- Clear navigation is important so users know where they are and what to do next

**Good example**

- When you print a document, initially it shows basic options like “How many copies?” and only displays two-sided printing, scaling, etc., if you press the “Advanced options” button.
- For a sign‑up form, have users enter just email and password first; fields like address or hobbies appear only after pressing “Enter more information.”

---

### 7) Accessibility

- It’s important that everyone can use the website: people with impaired vision, those who can’t distinguish colors well, people with limited hand mobility.
- Refer to the Web Content Accessibility Guidelines (WCAG) and consider alt text, keyboard navigation, etc.

**Good example**

- Use high contrast between text and background so it’s easy to read, and add descriptions (alt text) to images so screen readers can describe them.
- Allow navigation via keyboard for those who have difficulty using a mouse.

---

### 8) Proximity

- Place related elements close together so they’re perceived as a group, and separate unrelated elements

**Good example**

- On a video player, the play, fast-forward and rewind buttons are on the same row, while the stop/exit button is elsewhere.

---

### 9) Alignment

- If you place text and images randomly it looks messy
- Align elements along invisible lines (a grid) so things line up to the left or center; it makes content much nicer and easier to read.

<br/>

---

## 5) Six Core Principles Where UX + UI Meet (Simple Terms)

- **Clarity:** Make it obvious what to do at any moment
- **Consistency:** The same action should always look and act the same
- **Feedback:** Provide a reaction when something is tapped (loading indicators, toast messages, state changes)
- **Error prevention:** Make it hard to make mistakes (numeric-only fields, previews, confirmations)
- **Efficiency:** Reduce the number of steps to reach the goal
- **Accessibility:** Make it easy for everyone to see and interact (text size, contrast, keyboard use, etc.)

---

## 6) Super Simple Summary

- **UI = What you see (design components)**
- **UX = The overall experience when using it**
- Good UI helps create good UX,
- And good UX makes people want to use it again.

# UX / UI 쉽게 이해하기

## 1) UX랑 UI 한 문장으로

### UI (User Interface) = “화면 생김새 + 조작하는 버튼들”

- 버튼, 글씨, 색, 아이콘, 메뉴, 입력칸, 카드, 여백, 애니메이션 등
- **눈에 보이고 손으로 누르는 것들**

### UX (User Experience) = “쓰면서 느끼는 전체 경험”

- “쉽다/어렵다”, “빠르다/느리다”, “편하다/짜증난다”, “믿음 간다/불안하다”
- **사용하면서 생기는 기분 + 효율**

✅ **비유로 이해하기**

- UI = 자동차의 **운전대/버튼/계기판**
- UX = 자동차를 운전할 때 느끼는 **편함/안전함/스트레스**

> UI가 예뻐도 UX가 나쁠 수 있고, UX가 좋아도 UI가 별로면 쓰기 힘들어질 수 있어요.

---

## 2) UX가 왜 중요할까? (예시)

### 예시 A: 편의점 키오스크

- UI는 깔끔하고 예쁜데
- “취소” 버튼이 너무 작고 구석에 숨어 있음  
  → 사용자가 당황하고 짜증남  
  → **UX 나쁨**

### 예시 B: 배달앱

- 주소 입력 과정이 너무 복잡해서 단계가 많음  
  → “귀찮아” 하고 앱을 꺼버림  
  → **UX 나쁨(이탈)**

---

## 3) UX의 기초 구성요소 8가지 (쉽게)

### 1) 연구 & 공감

- 사용자 인터뷰, 설문조사, 관찰을 통해 사용자의 욕구/문제점을 파악

### 2) 문제 정의 (무슨 문제를 풀 건지)

- 연구 결과를 정리하여 사용자가 어떤 불편을 겪는지, 문제/목표가 뭔지 정하는 단계

**예시**

- “마트 가격 비교를 하려면 카톡방/전단지/말로 듣기 등 너무 번거롭다”
- 목표: “30초 안에 최저가를 찾게 해주자”

---

### 3) 사용자 이해 (누가 쓰는지 알기)

- 사용자(누가), 상황(언제/어디서), 동기/불안 요소를 파악하기

**예시**

- 유학생/이민자: 영어가 부담될 수 있음 → 쉬운 단어, 한글 지원
- 바쁜 사람: 단계가 많으면 중간에 나감 → 핵심 기능을 빨리 접근하게

---

### 4) 정보 구조(IA) (메뉴/탭 정리)

- 앱을 “책장”이라고 생각하면, IA는 “책을 어디에 꽂을지” 정하는 것

**좋은 예**

- Deals / Gas / Grocery / Profile  
  → 딱 보면 “어디에 뭐가 있는지” 감이 옴

**나쁜 예**

- 생활정보 / 정보공유 / 꿀팁 / 생활꿀팁 / 도움  
  → 비슷한 메뉴가 많아서 “어디 들어가야 하지?” 혼란

---

### 5) 사용자 흐름(User Flow) (목표까지 가는 길)

- 사용자가 목표를 이루는 길이 **짧고 자연스러운지** 확인하기

**예시: “가격 확인” 흐름**

- 홈 → 카테고리(마트) → 품목 검색(계란) → 최저가 보기

**나쁜 흐름 예**

- 홈 → 로그인 강제 → 약관 동의 여러 번 → 프로필 설정 → 검색 가능  
  → 중간에 사람들이 많이 나감

---

### 6) 와이어프레임 & 프로토타입 (스케치 → 눌러보는 시제품)

- 와이어프레임: 화면을 “연필로 그린 설계도”
- 프로토타입: 실제처럼 “눌러지는 시제품”

**예시**

- 버튼 위치/메뉴 구조를 먼저 대충 잡아보고
- 사람들이 헤매면 빠르게 수정

---

### 7) 사용성(Usability) (처음 써도 쉬운가?)

- 처음 쓰는 사람도 바로 할 수 있어야 함

**좋은 예**

- 회원가입 없이도 둘러보기 가능
- 입력 실수하면 친절하게 안내
  - “가격은 숫자만 입력해주세요”

---

### 8) 검증(테스트) (실제로 써보게 해서 문제/개선점 찾기)

- 만든 사람은 “당연히 보이는데?”라고 생각하지만
- 다른 사람은 “어디 있는지 모르겠는데?”라고 느낄 수 있음

**예시**

- 5명만 테스트해도 “막히는 지점”이 많이 나옴
- 진짜 사용 상황에서 관찰: “어디서 멈추는지”

---

## 4) UI의 기초 구성요소 9가지 (쉽게)

### 1) 레이아웃 & 우선순위(Visual Hierarchy)

- 중요한 게 먼저 보이게 만드는 것 (크기/굵기/색/여백)

**예시**

- “최저가 보기”가 가장 중요하면  
  → 버튼을 크게, 눈에 띄는 위치에

**나쁜 예**

- 중요한 버튼이 작고 회색이고 아래 구석에 있음  
  → 사람들이 못 봄

---

### 2) 글씨(타이포그래피)

- 글씨가 읽기 편해야 함

**예시**

- 글씨가 너무 작으면 눈이 아픔
- 줄 간격이 너무 좁으면 답답함

---

### 3) 색과 대비

- 색은 예쁘기만 한 게 아니라 “의미”를 전달함

**예시**

- 빨강: 삭제/위험/오류
- 초록: 성공/완료
- 회색: 비활성(지금 못 누름)

**나쁜 예**

- 버튼 색이 배경과 비슷해서 안 보임

---

### 4) 컴포넌트 일관성 (규칙 통일)

- 버튼/입력창/카드가 화면마다 다르면 사용자 혼란

**좋은 예**

- “저장” 버튼은 어디서든 같은 모양/색/크기  
  → 사용자가 빨리 익힘

**나쁜 예**

- A화면 저장 버튼은 파랑
- B화면 저장 버튼은 초록
- C화면 저장 버튼은 테두리만 있음  
  → “이게 같은 기능 맞아?” 혼란

---

### 5) 상태 디자인 (눌림/로딩/에러 등)

- 눌렀는데 반응이 없으면 사용자는 불안해함  
  → 다시 누름(연타) → 오류/중복 결제/중복 요청 위험

**좋은 예**

- 버튼 누르면 로딩 표시
- 완료되면 “저장 완료!” 메시지
- 실패하면 “네트워크 오류. 다시 시도해주세요.”

---

### 6) 점진적 공개(Progressive Disclosure)

- 한 번에 모든 정보를 보여주지 않고 단계적으로 필요한 만큼만 제공하여 사용자의 인지 부담 줄임
- 각 단계에서 어디에 있고 무엇을 해야하는지 알려주는 내비게이션이 중요

**좋은 예**

- 컴퓨터에서 문서를 인쇄할 때 처음에는 “몇 장을 인쇄할까요?” 같은 기본 옵션만 보여주고, “고급 옵션” 버튼을 누르면 양면 인쇄, 확대/축소 같은 자세한 설정이 나타남
- 회원가입 폼도 이메일과 비밀번호만 입력하게 하고, “추가 정보 입력”을 눌러야 주소나 취미를 입력하도록 설계

---

### 7) 접근성(Accessibility)

- 모든 사람이 웹사이트를 사용할 수 있게 만드는 것이 중요함. 시력이 약한 사람, 색을 잘 구분하지 못하는 사람, 손이 불편한 사람도 사용하기 쉽게 만들어야 함.
- WCAG(Web Content Accessibility Guidelines)를 참고하여 대체 텍스트, 키보드 내비게이션 등을 고려

**좋은 예**

- 글자 색과 배경 색의 대비를 크게 해서 잘 보이게 하고, 이미지에는 화면낭독기가 읽어줄 수 있는 설명(대체 텍스트)을 넣음
- 마우스를 쓰기 힘든 사람을 위해 키보드만으로도 메뉴를 이동할 수 있게 하면 좋음

---

### 8) 근접(Proximity)

- 관련 있는 요소를 가까이 배치해 그룹으로 인식하게 하고, 관련없는 요소는 떨어뜨려 배치

**좋은 예**

- 영상 플레이어를 보면 재생, 빨리감기, 뒤로가기 버튼이 같은 줄에 붙어 있고, 영상 종료 버튼은 떨어진 곳에 있다.

---

### 9) 정렬(Alignment)

- 글과 그림을 아무렇게나 배치하면 지저분해 보임
- 보이지 않는 줄(그리드)에 맞춰 왼쪽이나 가운데로 가지런히 정렬하면 훨씬 보기 좋고 읽기 쉬움.

## 5) UX + UI가 만나는 핵심 원칙 6개 (쉬운 말)

- **명확성:** 지금 뭘 해야 하는지 한눈에
- **일관성:** 같은 행동이면 항상 같은 결과/모양
- **피드백:** 눌렀으면 반응(로딩, 토스트, 상태 변화)
- **오류 예방:** 실수하기 어렵게(숫자만 입력, 미리보기, 확인)
- **효율:** 목표까지 단계 줄이기
- **접근성:** 누구나 보기/누르기 편하게(글씨 크기, 대비, 키보드 사용 등)

---

## 6) 초간단 정리

- **UI = 보이는 것(디자인 부품)**
- **UX = 사용해보고 느끼는 전체 경험**
- 좋은 UI는 좋은 UX를 돕고,
- 좋은 UX는 “다시 쓰고 싶다”를 만들어요.
